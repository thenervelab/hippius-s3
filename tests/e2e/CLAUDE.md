# tests/e2e/

End-to-end tests against a real hippius-s3 stack with mocked **backends**. Full API + gateway + writer + reader + workers stack runs in docker-compose; Arion/KMS/Hippius-API are replaced by mocks.

## Stack

[docker-compose.e2e.yml](../../docker-compose.e2e.yml) layers on top of [docker-compose.yml](../../docker-compose.yml) and adds:

- **mock-arion** — stands in for Arion storage. Deterministic identifier generation.
- **mock-kms** — OVH KMS mock on port 8443 (mTLS terminated locally).
- **mock-hippius-api** — blockchain/API mock. Hardcodes all tokens as `token_type="master"` — if you're testing sub-token behavior, use unit tests instead.
- **toxiproxy** — fault injection proxy. Pauses/delays/truncates traffic to selected services for resilience testing.

## Running

```bash
# Bring up the e2e stack
COMPOSE_PROJECT_NAME=hippius-e2e docker compose -f docker-compose.yml -f docker-compose.e2e.yml up -d --wait

# Run tests
pytest tests/e2e -v

# Tear down (includes -v to drop named volumes)
COMPOSE_PROJECT_NAME=hippius-e2e docker compose -f docker-compose.yml -f docker-compose.e2e.yml down -v
```

Env: [.env.test-docker](../../.env.test-docker) — uses docker service names (e.g. `http://api:8000` for the API, `mock-kms:8443` for KMS).

## Conventions

- **Mock hippius source header** ([conftest.py:59-71](conftest.py)): when NOT running against real AWS, assert the `x-hippius-source` response header to verify the request hit our stack and not some upstream.
- **Skip markers** ([conftest.py:37-51](conftest.py)):
  - `@pytest.mark.s4` — S4 append tests.
  - `@pytest.mark.local` — skipped when `RUN_REAL_AWS=1`.
  - `@pytest.mark.hippius_cache`, `@pytest.mark.hippius_headers` — skipped against real AWS.
- **Real AWS mode**: `RUN_REAL_AWS=1` or `AWS=1` — swaps boto3 for real AWS S3 to verify wire compatibility. Assumes proper AWS credentials in env.
- Fixed test seed phrase and access key fixtures for reproducible SigV4.

## Gotchas

- **FS cache is a shared volume across all services** in docker-compose — if one service pollutes it, subsequent tests can see stale state. Use the cache-clear helpers but note that DLQ requeue tests intentionally preserve cache state (see commit `237f143`).
- **toxiproxy restart** resets all chaos rules — be sure the test's setup reconfigures what it needs.
- **`clear_object_cache` runs `rm` inside the api container** (see commit `f8fbc23`) so permissions match the runtime user.
- **Test isolation**: Bucket names should include a UUID or per-test prefix; leftover buckets from a failed test will poison subsequent runs.
