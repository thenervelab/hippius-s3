## 1. Why
s3-2.0 introduced the tiered cache: node-local SSD ingest → CephFS via the new Rust **`hippius-drain`** stack
(`drain-allocator` singleton + `drain-agent` DaemonSet), with the api fleet running all-local on staging. Two
inputs drive s3-2.1:
- **The architecture-review audit** of `hippius-drain` (findings **F1–F22**; safe-to-run in staging, no data-loss
  bug, but 3 P1s + many P2s/nits).
- **Our own validation on staging (2026-06-18):** the drain is **dead-stalled** (every `cephor_replication_status`
  row `pending`, agents looping `"draining a part failed"`, 0 replicated), and the **enqueue-timing race** is real
  (api enqueues the Arion upload at PUT time → uploader blocks 30s polling ceph holding its inflight slot →
  head-of-line stall → DLQ churn; reconcile ~60s > 30s wait).

**Goal:** a clean, releasable s3-2.1 that makes the drain architecture *correct and production-bound*: green the
drain, fix the enqueue-timing race (drain-gated upload), enforce CI/lint/supply-chain + safety guardrails, and
validate end-to-end with a benchmark.

## §0. ARCHITECTURE CHANGE (2026-06-24) — drain-direct enqueue, delete the promoter

**What shipped (PR #196):** the drain-gated upload as "Mode B" — the Rust drain emits
`pg_notify('cephor_replicated', …)` on `mark_replicated`; a Python **upload-promoter** worker `LISTEN`s +
sweeps + rebuilds the `UploadChainRequest` + calls `enqueue_upload_to_backends`. Plus PR-7a (uploader
defer-gate, live) and an `object_versions.address` column. Validated working (1GB: arion 256/256 + ovh
256/256, zero DLQ).

**Why rework:** the promoter + `pg_notify` + sweep + defer-gate is a lot of machinery for "when a part lands
on ceph, push an upload request." Lean it.

**New design — drain-direct (hard cutover, NO flag):** the **Rust drain enqueues the upload request to
Redis directly**, the moment a part replicates. The api stops enqueuing uploads entirely (always writes
`object_versions.address`, never enqueues). The drain becomes the **sole producer** of new upload requests.
- **At-least-once, no sweep:** LPUSH **before** the `mark_replicated` commit → a crash redrains+re-LPUSHes
  (dup-safe via the idempotent uploader: `skip_if_exists` + `chunk_backend ON CONFLICT`).
- **No duplicated logic:** Python no longer produces upload requests for new objects, so it's a single
  producer→consumer wire contract (uploader still deserializes `UploadChainRequest`; `extra="ignore"`+defaults
  make it forgiving). The retry/DLQ/ops-script producers (`dlq_requeue.py` etc.) stay in Python unchanged.
- **Trigger unchanged:** kept the reconciler **scan** (it's the durable backstop; a copy-request queue was
  evaluated + rejected as additive machinery — Redis isn't atomic with the FS write, so the scan stays anyway).
- **Rollback:** code revert + redeploy (no flag — hard cutover, owner's call).

**Delete:** `upload_promoter.py` + `run_upload_promoter_in_loop.py` + its test; `promoter_build_request.sql` +
`promoter_sweep_unpromoted.sql`; `k8s/staging/upload-promoter-deployment.yaml` + kustomization line; config
flags (`drain_gated_upload_enabled`, `promoter_sweep_*`); the `pg_notify` CTE in `mark_replicated` + its test;
the PR-7a defer-gate (`all_parts_on_pool`, `defer_upload_request`, `move_due_deferred`, the mover,
`uploader_not_ready_*` config, defer tests).

**New (Rust):** async `redis` dep; agent config `REDIS_QUEUES_URL` + `HIPPIUS_UPLOAD_BACKENDS`; an
`UploadEnqueuer` trait seam (core stays Redis-free) called in `drain_part` before `mark_replicated`;
`Store::load_upload_context` (objects⋈object_versions⋈buckets + latest `multipart_uploads.upload_id`); a Rust
`UploadChainRequest` serde producer fanning out to `{backend}_upload_requests`.

**Keep:** `object_versions.address` (+ migration/wrapper), `claim_seq` fencing (PR-2), drain concurrency
(PR-8), node-isolation (PR-6), `queue.py` consumer/retry/ops producers.

**Companion (separate `s3-backup` PR):** remove that worker's defer-gate (the PR #15 mirror) — follow-up.

**Risk:** drain is now a hard dependency for all uploads (data still safe on SSD/ceph; janitor won't evict
un-replicated). Wire-contract coupling Rust→Python (small/stable model; pydantic-forgiving).

### What actually shipped in PR #199 (drain-direct + colleague hardening, 2026-06-24/25)
Beyond the base cutover (drain enqueues, api writes address, promoter/notify/sweep/defer-gate deleted), the
PR grew the supporting machinery the cutover needs to be safe — **all CI green incl. e2e**:
- **Deferral/backoff (not just "defer once"):** `cephor_replication_status.deferred_until` (migration **0008**);
  the drain releases + backs off a not-ready part (address not written yet) so not-ready parts don't starve
  ready ones and don't trip the **Ceph circuit breaker** (`BreakerSignal::Deferred` vs `CephFailure` — a real
  Ceph IO error still trips it, a deferral doesn't). `claim_part` skips `deferred_until > now()`.
- **MPU reaper** (`hippius_s3/services/mpu_cleanup.py` + `workers/run_mpu_reaper_in_loop.py` +
  `k8s/staging/mpu-reaper-deployment.yaml`, replacing the promoter deployment): the terminal path for
  **abandoned/aborted MPUs**. A version whose `address` was never written + initiated >`mpu_stale_seconds`
  (1 day) ago → mark its `cephor_replication_status` rows `'failed'` (`fail_replication_status_for_version.sql`,
  `list_abandoned_versions.sql`) so the drain stops re-claim/re-defer churn fleet-wide. Without it the deferral
  loops forever. Also wired into the **abort** path (synchronous fail + best-effort local cache delete).
- **Reader fix:** `DualFileSystemPartsStore.chunks_exist_batch` now consults the pool fallback — drained parts
  that live only in the pool dir were wrongly routed to the download pipeline.
- **DLQ requeue fix:** `dlq_requeue.py` now requeues **every** per-part DLQ entry, not just the first.
- **Wire-contract golden test:** `tests/fixtures/upload_chain_request.golden.json` pinned from **both** sides
  (Rust producer test + Python `model_validate` test) → cross-language drift breaks CI.
- **`load_upload_context` version-scoped** `upload_id` subquery (no cross-version `simple::`↔`multipart::` leak).
- **e2e:** the real drain stack (`drain-allocator` + `drain-agent`) is now in `docker-compose.e2e.yml` with
  SSD-ingest→pool topology + REDIS_QUEUES_URL/HIPPIUS_UPLOAD_BACKENDS parity; unlimited drain rate in e2e only.
- **Runbook:** `docs/drain-direct-rollout.md` (deploy-drain-first ordering, re-drive SQL, hard-cutover/rollback).

### Review findings (2026-06-25, 4 parallel reviewers) — fixed + tracked
- **FIXED (P1/P2): abort + complete used `objects.current_object_version`** instead of the upload's own
  `parts.object_version`. A concurrent same-key MPU advances that pointer, so the earlier upload's abort would
  `fail_version_replication` + `delete_object` a *different, possibly in-flight* version (cross-upload
  corruption), and complete/`set_object_version_address` would target the wrong version. Fixed by deriving the
  version from the upload's own parts (`get_multipart_version_by_upload.sql`) in both paths. (read-only
  `list_parts_internal` left as-is — non-destructive, pre-existing.)
- **DISMISSED:** dlq_requeue `break`-on-permanent (deliberate "refuse without `--force`" safety semantic).

### Still-relevant remaining tasks
- **Append double-produce (deferred, owner-OK):** `extensions/append.py` still calls `writer_enqueue_upload`,
  so an appended part is enqueued by the api AND (on replication) by the drain. Harmless today (idempotent
  uploader dedups) but breaks the "drain is sole producer" invariant — route append through `set_object_version_address`
  when append is next touched.
- **Orphan GC for abandoned MPUs (load-bearing follow-up):** the reaper marks replication `'failed'` but the
  janitor's gate keys on `chunk_backend` (a different table), so abandoned parts' SSD+ceph bytes are never
  evicted → permanent leak. Abort only cleans the one node it hit. Track leaked-bytes; build the orphan GC.
- **Reaper integration test + `multipart_uploads.initiated_at` index** — unit test uses a SQL-blind FakeDb;
  add a real-PG test asserting `list_abandoned_versions.sql` excludes completed/address-bearing/fresh uploads.
- **Observability:** export a `cephor_drain_deferred` metric (the counter exists but is only used to keep
  deferrals out of `error_bps`) so the cutover + reaper-lag are watchable. Second golden fixture for the
  simple-PUT shape (current golden always has `upload_id`).
- **PR-3b** — F9 lints (`arithmetic_side_effects` + `indexing_slicing`), 39 sites measured, deferred.
- **s3-backup** — remove the OVH worker defer-gate (companion to §0).
- **§8 infra** — staging Postgres-on-CephFS slow (prod PG is on NVMe; staging is for correctness only).
- **PR-10** — formal ×3 throughput benchmark from a co-located client (staging numbers are client/PG-bound).

### New finding from PR-1 validation (add to a PR)
- **G2 — reconciler doesn't adopt existing `pending` rows.** The migration's one-time DELETE only clears rows present *at migrate time*; rows created afterward by a pre-node-scoping agent (or any NULL-node `pending` row) are never adopted, since `reconcile_parts` only `record_landed`s `status=None` parts. Self-heals only via a manual `DELETE … WHERE node_id IS NULL`. **Durable fix (fold into PR-2 or a small PR):** have the reconciler `record_landed` (idempotent UPSERT) for `Some(Pending)` nodeless rows too, OR make `status()` treat a NULL-node row as absent. Avoid per-cycle writes on the slow staging PG → prefer adopting only when `node_id IS NULL`.

## 4. Workstreams

### PR-1 — Drain node-scoped claim (G1) + rate-control (F1/F7) + observability  [Rust, P1 BLOCKER]
**G1 (NEW, confirmed root cause of the live stall) — global claim queue vs node-local SSD.**
`cephor_replication_status` has **no `node_id`** (migration 0005); `record_landed_part` (store.rs:518) records none
and `claim_part` (store.rs:544) is global (`WHERE status='pending' … SKIP LOCKED LIMIT 1`). But each `drain-agent`
only mounts its **own** node's SSD (node-local hostPath). So an agent claims parts whose data is on the *other*
node → `list_chunk_indices` returns `Ok(empty)` for the missing-local dir (localfs.rs, by design) → `drain_part`
skips to `persist_meta` → `create_dir_all`s the empty pool dir then `copy`s a non-existent local meta → **ENOENT →
`Io(Persist)` → the generic `"draining a part failed"` WARN.** Net: agents waste cycles on foreign parts, ~nothing
drains. **Fix:** add `node_id` to `cephor_replication_status` (migration 0006); reconciler stamps its
`CEPHOR_NODE_ID` via `record_landed` UPSERT (`ON CONFLICT … DO UPDATE SET node_id=EXCLUDED.node_id WHERE node_id IS
NULL`, self-healing existing NULL rows); `claim_part(node_id)` filters `AND node_id=$node_id` (incl. the
stale-`draining` reclaim); update the `pending` partial index → `(node_id, landed_at)`. Tests: an agent must not
claim a foreign-node part; a part is claimed only by the node that recorded it.
**Plus the rate-control + observability items:**
- **F1** (`crates/hippius-drain-agent/src/worker.rs:107`, `crates/hippius-drain-core/src/enforce.rs:172` `try_take`,
  `agent/src/main.rs:39` burst=max_drain_rate): a part with bytes > burst can never be admitted → loops
  Denied→release→re-claim forever. Fix: clamp charge to burst (`try_take(min(bytes,burst))`) + reconcile remainder,
  or one-shot overdraft when a single part > burst; or validate `burst >= max_part_size`. **Regression test:** part
  with `bytes > burst` must drain, not loop.
- **F7** (`agent/src/worker.rs:72` `part_size`): stat/list failure charges 0 → admits unmetered. Fix: on stat error
  **deny** admission (charge `burst`/over-budget) and retry next wake; or meter actual bytes post-transfer.
- **Live stall:** set `RUST_LOG=debug` on one agent, capture the concrete error behind `"draining a part failed"`;
  inspect `cephor_replication_status` failed rows; check staging-PG latency. Fix root cause (likely raise the
  allocator static ceiling — per-node budget was ~528 KB — and/or relieve PG contention).
- **F14 (nit, decide here):** `DRAIN_CONCURRENCY=4` is unused (serial drains). Decide serial vs batched-claim.
- **Acceptance:** staging `cephor_replication_status` moves pending→replicated; 100 MB+ object drains; no
  `"draining a part failed"` loop; uploader stops DLQ-churning on drained objects.

### PR-7 — Enqueue-timing fix: drain-gated upload  [Python + 1-line Rust, P1 functional — HEADLINE]
**EMPIRICALLY PROVEN on staging 2026-06-19 (1GB MPU, object `ce2815ef`):** MPU completed 08:44:05 → api enqueued
arion upload immediately → uploader blocked in `get_meta_with_wait` and **DLQ'd at 08:44:36 (`permanent`, "Missing
meta in filesystem cache after 30.0s … part=1")**. The drain didn't finish copying that object to ceph until
**08:47:01** — i.e. the 30s uploader wait expired **~2.5 min** before the data was actually on ceph. Result:
`object_version.status=failed`, **arion 0/256 chunks, ovh 2/256 chunks** persisted, yet the object still GETs fine
(md5 exact) because the read path serves from the CephFS cache. **Downstream risk beyond "failed":** no
`chunk_backend` arion row ⇒ the janitor's replication gate keeps those parts on ceph **forever** (can't evict
un-replicated chunks) ⇒ ceph cache slowly fills. Current staging DLQ depths from this race: **arion 5281, ipfs
7781, ovh 265**. This is the single highest-impact fix in s3-2.1.
**Problem (validated):** api enqueues `arion_upload_requests` at PUT/MPU-complete before the part is on ceph;
`arion-uploader` (`MAX_INFLIGHT=4`/pod) then blocks up to 30s in `get_meta_with_wait` holding its slot → head-of-line
stall → DLQ. **Verified enablers:** the upload workers are DB-driven by part identity (only `address` is
non-derivable); multi-backend fan-out already exists (`enqueue_upload_to_backends`, `hippius_s3/queue.py`);
per-backend persistence already tracked in `chunk_backend`; drained-to-ceph already tracked in
`cephor_replication_status.status='replicated'`.
- **Fix (event-driven, minimal Rust):**
  1. Migration: add `address` (main account) to `object_versions` (`hippius_s3/sql/migrations/`), api writes it in
     `object_writer.py`.
  2. Rust (1 line, with PR-2): `pg_notify('cephor_replicated','<object_id>:<version>:<part>')` in `mark_replicated`.
  3. New `hippius_s3/workers/upload_promoter.py` + `workers/run_upload_promoter_in_loop.py`: `LISTEN cephor_replicated`,
     build the full `UploadChainRequest` from DB by object_id, call existing `enqueue_upload_to_backends()`.
  4. Flag-gate the api PUT/MPU enqueue (`put_object_endpoint.py`, `multipart.py`) to flip api-enqueue ↔ promoter.
  5. `k8s/staging/upload-promoter-deployment.yaml` + kustomization entry.
- **Interim mitigation (ship first, can be its own micro-PR):** uploader requeue-with-delay instead of a 30s blocking
  poll — quick `cephor_replication_status`/ceph-meta check; if absent, push to a delayed retry set + release the
  inflight slot. Removes head-of-line blocking immediately.
- **Acceptance:** uploads fire only post-ceph; uploader DLQ ~0; no inflight saturation; per-part pipelining on MPU.

### PR-8 — Reconciler hardening  [Rust, nits]
**Findings:** F15 (paginate `scan_parts`/status round-trips), F16 (TOCTOU `recovered` over-count), F17 (post-replication
S4 appends not re-drained — **design decision** for owners), F19 (backlog metric assumes dedicated ingest volume).
- **F14 batching (promoted from nit — now a perf lever, see §8).** PG stays on CephFS, where every `COMMIT` pays a
  ~22 ms–2 s `fdatasync`. The drain currently commits **per part, serially** (claim + `mark_replicated` in separate
  txns; `DRAIN_CONCURRENCY` unused), so a 128-part object pays O(256) fsyncs → the slow replication tail. Fix:
  claim N parts per txn + batch `mark_replicated`, and/or honor `DRAIN_CONCURRENCY`. Biggest code-side win on lag.
- **Allocator hot-row churn (see §8).** `cephor_leader_lease`/`cephor_node_state`/`cephor_allocation` UPSERT every
  ~2 s tick → constant dead tuples + autovacuum + fsync on the slow volume. Skip no-op writes (`WHERE` guard) or
  lengthen cadence.

## 8. Known staging infra issues (not s3-2.1 code, but blocking validation)
- **Staging Postgres is slow — ROOT-CAUSED 2026-06-19: WAL `fdatasync` on CephFS, NOT a missing index.** Staging
  runs the **ceph-backed `postgres` CNPG cluster** (`postgres-1`, data PVC on storage class `ceph-filesystem`).
  Symptom: drain `claim`/`mark_replicated` and allocator `cephor_leader_lease`/`cephor_node_state` statements WARN at
  **1–2.8 s** (sqlx `slow_threshold=1s`), inflating the per-part drain (the slow 5-part tail on the 1GB test).
  - **DECISION (owner, 2026-06-19): Postgres STAYS on Ceph. Do NOT move to NVMe.** So treat the fsync latency as a
    fixed cost of the environment and minimise how often the *code* pays it.
  - **Evidence the slowness is fsync, not query/index/bloat/CPU/replication:**
    - A **bare `COMMIT`** (no table, no index) takes 1.08–2.3 s in the logs → time is in WAL flush, not planning.
    - `pg_test_fsync` on the `ceph-filesystem` pgdata volume: `fdatasync` = **46 ops/s ≈ 21.7 ms/op** (NVMe is
      ~5–10k ops/s ≈ 0.1–0.2 ms → CephFS is **~50–100× slower**); worst pattern (`write,close,fsync`) = **539 ms/op
      unloaded**. Under shared-Ceph contention the tail stretches to the 1–2.8 s commits observed.
    - `synchronous_commit=on`, `wal_sync_method=fdatasync`, `synchronous_standby_names` **empty**, `postgres-2` is an
      **async** replica → commits do NOT wait on a replica; the latency is purely local WAL fsync to CephFS.
  - **Ruled out (the "is it an index?" question):** `cephor_replication_status` already has the PK + the
    `..._pending (node_id, landed_at) WHERE status='pending'` index (PR-1). `EXPLAIN` on `claim_part` shows a **Seq
    Scan**, but that is *correct* — the table is ~170 live rows (1 page), so a scan beats an index descent; an index
    cannot help, and `COMMIT` latency is index-independent regardless.
  - **Ruled out:** CPU throttling (limit 4 cores, ~23 s throttled over 88 d, idle at 6m); checkpoint pressure
    (`pg_stat_checkpointer`: 6630 timed vs **3** requested); synchronous replication (async, see above).
  - **Secondary / additive (worth a cheap fix):** hot-row churn. `cephor_leader_lease` = 1 live / **37 dead** tuples,
    `cephor_allocation` autovacuumed **1124×**, `cephor_node_state` 227× — the lease + node_state + allocation rows
    are UPSERTed every ~2 s tick, so autovacuum runs almost continuously, adding background WAL+fsync IO to the same
    slow volume. Tiny in row terms (doesn't slow scans) but it's extra fsync traffic competing with the drain.
  - **What WE can do given PG-stays-on-Ceph (code-side, owned by us):**
    1. **Batch the drain's DB writes (F14, fold into PR-8).** Today the drain commits multiple times per part (claim
       in one txn, `mark_replicated` in another), **serially** — `DRAIN_CONCURRENCY` is unused. Each commit = one
       CephFS fsync. Claim N parts per txn + batch `mark_replicated` (and/or run the configured concurrency) so
       128 parts cost O(few) fsyncs instead of O(256). **This is the single biggest code lever on the slow tail.**
    2. **Slow the allocator's hot-row cadence** (or `UPDATE … WHERE` to skip no-op writes) so the lease/node_state
       UPSERT every 2 s isn't generating a dead tuple + fsync per tick when nothing changed.
    3. **Consider `synchronous_commit=off` for the drain's session/role only** (staging) — trades a sub-second
       crash-loss window for not blocking each commit on a CephFS fsync. Acceptable for the replication-status
       bookkeeping on staging; **never** blanket-set on prod. (Open: is the drain's DB write loss-tolerant enough?)
  - **Caveat for PR-10:** any replication-lag/throughput numbers measured on staging are **PG-fsync-bound**, not real
    drain throughput. Note this in the report; the prod NVMe cluster will not show the slow tail.
