# s3-2.1 — drain hardening & production readiness

> Living TODO for the Rust `hippius-drain` stack (lives on the `staging` branch). One surgical PR per work item.

## Architecture (current)

Node-local SSD ingest → CephFS pool via the Rust hippius-drain stack (`drain-allocator` singleton + `drain-agent`
DaemonSet). The API writes each part to node-local SSD + `object_versions.address`; it **never enqueues uploads**.
The **drain is the sole upload producer**: once a part replicates to the pool it LPUSHes an `UploadChainRequest` to
Redis; the uploader ships it to the storage backend and records `chunk_backend`. State machine:
`cephor_replication_status` = `pending → draining → replicated | failed` (terminal sinks). The janitor evicts a
CephFS part only once every required backend has a live `chunk_backend` row — or once the part is terminally
abandoned.

The cutover is **flagless**: the drain is the sole upload producer with no fallback, so every availability item
below is live today, not a latent prod-only risk.

**Priority note:** prod Postgres runs on dedicated NVMe; prod adds dedicated SSD nodes for ingest. So the PG-relief
track is churn-reduction (not a cliff), and SSD cleanup is required (a full SSD 503s every PUT on that node).

## Work items

### P1 — required before production
- **WI-1 — completeness check before commit.** `partdrain.rs` commits `Replicated` + unlinks the SSD source
  trusting `list_chunks`, without parsing `meta.json`'s `num_chunks`, so a part with meta present but an interior
  chunk missing is committed and the only good copy deleted. Add a completeness check (declared == present, no
  holes) → **defer, not hard-fail**. *(one PR with WI-2/3 — same `drain_part` region)*
- **WI-2 — verify every chunk.** The copy byte-verifies only the first + last chunk (sampled); interior chunks
  commit unverified with no post-`Replicated` re-drive. Full-readback every chunk.
- **WI-3 — guard + retry `mark_failed`.** `mark_failed` lacks the `claim_seq` fence `mark_replicated` has (a stale
  claimant can fail a live part), and a single transient mismatch terminally discards a healthy part. Add the guard
  + a bounded retry before failing.
- **WI-6 — decay-on-`Err`.** On a sustained Redis outage the agent holds its last budget forever and keeps draining
  at full rate. Decay toward floor on the `Err` arm like the TTL-expiry path (`runtime.rs`).
- **WI-8 — coordination Redis can evict its own fence.** The leader epoch counter is `INCR`ed with no TTL on an
  `allkeys-lru` Redis; if evicted it resets and re-issues spent epochs, mis-firing the fence → allocator deadlock.
  Interim: a Postgres epoch-floor (new migration). Permanent: a dedicated `noeviction` coordination Redis (Phase 7).
- **WI-16 — CI runs the coordination tests.** CI has Postgres but no Redis and never sets `CEPHOR_TEST_REDIS_URL`,
  so the fencing/election/lease/decay tests skip silently. Add a Redis service and run the tier.
- **WI-5 — liveness probe / health signal.** Neither binary exposes health and neither manifest has a
  `livenessProbe`, so a hung (not crashed) runtime stays `Running` = silent per-node upload outage. Add a
  heartbeat-file + exec probe on both.
- **WI-7 — non-zero exit on worker fault.** The agent returns `Ok(())` regardless of the run report, so a panicked
  worker exits 0 (no restart, no alert). Return a failure exit code on fault.
- **WI-4 — drain metrics → Grafana.** The drain exports zero metrics (tracing-only). OTLP-push to the collector
  (no `/metrics` server) with resource attributes so series land under the staging namespace filter. *(detail in
  `s3-2.1-rust-handoff.md`)*

### P2 — hardening
- **WI-9 — breaker `Probing` deadline.** The circuit-breaker `Probing` state has no deadline; a cancelled probe
  wedges it and denies all drains until restart.
- **WI-10 — breaker error domain.** A Postgres/claim error during commit currently maps to a Ceph failure and can
  open the node-global Ceph breaker on a healthy pool. Only ceph-write errors should trip it.
- **WI-11 — Redis response timeouts.** The `Coordinator`'s connection manager has no response timeout, so a
  half-open Redis hangs the tick/heartbeat/alloc-pull loops. Set response + connection timeouts.
- **WI-12 — backends from secret.** The drain hardcodes its upload-backend list in the manifest instead of sourcing
  the shared secret; adding a backend would silently under-replicate. Use the secret reference.
- **WI-14 — allocator-before-agent ordering.** The allocator owns migrations; the agent has no schema-wait
  initContainer, so agents CrashLoop on a fresh-namespace apply. Add a `wait-for-schema` initContainer.
- **WI-15 — ancestor-dir fsync.** `finalize_part` fsyncs only the part dir, not its ancestors; a crash after commit
  can strand the part (orphan-debris, re-drivable — not data loss). fsync the ancestor chain.
- **WI-13 — batch the reconciler query (R3).** The reconciler runs one status SELECT per scanned part per cycle
  (O(N)); batch into one `UNNEST` query.
- **WI-17 — terminal-row GC (R4).** `cephor_replication_status` never prunes terminal rows, bloating the hot claim/
  reconcile scans. Add a leader-side aged-row GC.

### P3 — polish & validation
- **WI-18 — resilience + hygiene (R5/R6).** defer-cap + metric; AIMD smoothing (EWMA, dead-band, safe floor);
  full-jitter backoff; worker readiness probes; lease revalidation on allocation writes; arithmetic/indexing lints;
  `O_NOFOLLOW`; non-root pods.
- **WI-19 — scale + chaos harness** (after WI-16): GB-scale MPU benchmark + replication-lag experiment, burst/
  SSD-fill, and a chaos matrix (kill node; Redis blip/eviction/hang; Ceph degradation) with an end-to-end checksum
  no-data-loss gate.

### Observability & infra (Python side, complements WI-4)
- Instrument the un-metered loops (MPU reaper first; orphan-checker, account-cacher, cachet, DLQ) via the shared
  monitoring collector, then build the "S3 Drain" Grafana dashboard consuming WI-4 + the reaper/janitor counters.
- Real-PG reaper integration test + `multipart_uploads.initiated_at` index; a co-located throughput benchmark.

## Edge auto-caching

Goal: auto-cache read traffic at the edge — **including private content** — to (a) bridge the cross-node window
between a fresh upload landing on one node's SSD and its replication to the shared pool (a GET routed to a different
node currently 503s until replication), and (b) hold a transient secondary copy during that window. The edge cache
is best-effort/evictable — never a durability guarantee; the read path must always fall back SSD → pool → backend.

Current state: the gateway side is built (cache-control policy, an auth-probe endpoint, and cache purge-on-write);
the edge configuration that uses them is not yet deployed, and content is private-by-default.

- **EC-1 — private-content auth probe.** Wire the edge auth-subrequest to the gateway's existing probe endpoint,
  plus a visibility demotion (store cacheable at the edge, but rewrite to `private, no-store` on egress so clients
  don't cache it). Security linchpin: authorize **every** request, including cache hits — otherwise cached private
  content is served unauthorized. Deploy + verify on staging first.
- **EC-2 — passive auto-cache.** With EC-1 live, any successful GET (public or private) populates the edge, so
  subsequent cross-node reads during the replication window are served from cache. (Active post-upload warming is a
  later option, only if residual first-read 503s prove material.)
- **EC-3 — cache-by-default flip (gated).** Only after EC-1 is proven: make new buckets cacheable by default
  (explicit + revertable), so the edge caches everything under the auth-probe gate.
- **EC-4 — edge config management.** Bring the edge fleet under config management and reconcile drift, so staging
  mirrors prod and changes apply uniformly.
- **Future:** if the per-request auth round-trip proves costly, migrate to edge-validated signed tokens/URLs (the
  CDN-standard model — presigned URLs already provide a token-like primitive).

## Path to production

Sequenced; each phase gates the next. Nothing ships to prod until the topology sign-off.

1. **Correctness (staging).** Land WI-1/2/3 (data integrity), WI-6 (decay-on-`Err`), WI-15 (ancestor fsync).
   *Gate:* a part is never committed `Replicated` unless every declared chunk is present and verified; a transient
   fault never discards a healthy part; Redis-outage decays to floor.
2. **Observability & operability.** WI-4 (metrics + dashboard) + instrument the Python loops; WI-5 (liveness) and
   WI-7 (exit-on-fault). *Gate:* throughput, replication lag, backlog, failures, coordinator health, and DLQ depths
   are visible; a hung or panicked pod restarts and alerts.
3. **Hardening & hygiene.** WI-8a (epoch-floor), WI-9/10/11 (breaker + Redis timeouts), WI-12/14 (secret + ordering),
   WI-13/17 (reconciler batch + terminal GC), WI-18. *Gate:* bounded PG churn; the fence survives Redis memory
   pressure; the breaker can't wedge or trip on non-Ceph faults.
4. **Scale & chaos validation.** WI-16 (CI coordination tests) as prerequisite, then WI-19. *Gate:* throughput
   acceptable, lag bounded, every failure self-recovers, and byte-for-byte integrity confirmed by checksum.
5. **Edge auto-caching (staging).** EC-1 (auth-probe + visibility demotion) and EC-4 (edge config management), then
   EC-2/EC-3. *Gate:* no private-data leak on a cache hit is proven; cross-node gap bridging works; per-hit auth
   latency acceptable.
6. **Prod readiness.** Rollout/rollback runbooks (drain ships before the API image that stops PUT-enqueue);
   alerting on drain lag, SSD pressure, DLQ growth, coordinator leader-loss/fence anomalies. *Gate:* runbooks +
   alerts rehearsed on staging.
7. **Topology decision meeting.** SSD ingest nodes (count/sizing vs lag + pressure), taints/isolation, scaling
   strategy (DaemonSet + singleton placement, graceful node add/drain), and the dedicated `noeviction` coordination
   Redis (WI-8b). *Gate:* signed-off topology + cutover runbook.
